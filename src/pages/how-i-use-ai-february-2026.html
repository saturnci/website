How I use AI to build features for SaturnCI (February 2026)

How AI changes programming for me

Programming involves a lot of sticking points. Learning a new technology is a sticking point. Looking up a bit of syntax that I can't remember is a sticking point. All these little sticking points slow me down. They can even sometimes make programming less fun.

AI is like a mental lubricant. It lets me glide right through these sticking points. Not all of them, of course, but a lot of them. For example, I never write tests by hand anymore. Why would I? I just tell Claude Code my high-level behavior specifications, let it figure out the syntax for me, and then make any necessary adjustments. Then I let Claude Code run the tests for me and then write the code to make the tests pass. It's a dream.

One of the ways AI helps me, naturally, is that it allows me to carry out my work faster. It also helps me work at "lower RPMs", doing the same work as before but will less mental exertion per unit of time. Just as important, and often more so, AI gives me the gumption to take on ambitious projects that I never even would have attempted in the pre-AI world because they seemed too hard or too time-consuming.

Even though AI was already quite useful in its "primitive form" when we had to tab back and forth between a chat window and the editor, the arrival of agentic tools like Claude Code was a distinct inflection point for me. Today I open my editor fairly seldom, and usually only to read a piece of code or make a minor tweak. I've found that whenever I want to make a code change, it's usually cheaper to tell Claude to make the change than to type it out myself. I rarely write code directly anymore; I prefer the higher level of abstraction that AI affords.

How AI doesn't change programming for me

Even though AI has radically transformed my day-to-day programming work, it obviously hasn't changed the natural facts of reality. To use an extreme example, the existence of AI doesn't mean that the benefits of version control have gone away, of course. I doubt many people would argue with me on that one. I think most other programming principles retain their benefits as well, such as atomic commits, small units of work, tight feedback loops, continuous integration/continuous delivery, single-piece flow, easily-changeable code, evolutionary design, refactoring, user research, usability testing, modularity, loose coupling, test-driven development and so on. We can get an illusion of higher productivity for a while by skipping these practices and just "vibe coding", but that will just make our blades get duller faster (messy code equals dull blade) and bring us very quickly to the point where our blades are too dull to cut with at all.

Test-driven development was at the center of my development workflow before AI and it still is now. Let's make a brief review of what TDD is and then look at how I practice TDD with AI.

How I TDD using Claude Code

Before I explain how I TDD with Claude Code I want to clarify what TDD actually is. Kent Beck wrote a blog post called Canon TDD which is about the closest thing to an "official" explanation of TDD that I know of. Here's the formula from Kent's post:

  1. Write a list of the test scenarios you want to cover
  2. Turn exactly one item on the list into an actual, concrete, runnable test
  3. Change the code to make the test (& all previous tests) pass (adding items to the list as you discover them)
  4. Optionally refactor to improve the implementation design
  5. Until the list is empty, go back to #2

If you're familiar with the red-green-refactor loop (write a failing test, make it pass, then optionally refactor) then you might recognize the algorithm above as basically the same thing. I personally prefer a certain different formulation of the TDD process, one you can think of as being spiritually the same thing as red-green-refactor, just "shot from a different angle". I call it the specify-encode-fulfill loop, and the reason I prefer it is that I think a) it's easier to explain and b) it's more self-evidently compelling. It goes like this:

  1. Come up with the specifications for what you want to build
  2. Encode those specifications as automated tests (aka executable specifications)
  3. Write the code to fulfill the specifications

Since I believe specify-encode-fulfill to be easier than red-green-refactor for humans to grasp, I figure it should be easier for machines to grasp as well. Below is how I present TDD and specify-encode-fulfill to Claude Code.
