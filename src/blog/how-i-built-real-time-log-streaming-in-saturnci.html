How I Built Real-Time Log Streaming in SaturnCI

When you're watching a test suite run on SaturnCI (my continuous integration platform for RSpec and Rails), you can see the log output appear in real time, line by line, as the tests execute on a worker machine. The log output looks like this:

When your test suite is running on a CI platform, you want to be able to see your log output in real time. One of the first things I had to figure out when building SaturnCI, my continuous integration platform, was how to make this happen.

The architecture of SaturnCI is that tests get executed on ephemeral worker machines, which are separate from the web servers that host the SaturnCI web application. I decided to break down the log streaming challenge into two fundamental parts: One, how do I get log content from the workers to the web servers? Two, how do I get log content from the web server to the browser?

As usual, I decided to take a steel thread approach with this feature, meaning I wanted to get a rudimentary version of the feature working end-to-end as early as possible in order to minimize the amount of work-in-progress and minimize the risk of unproductively spinning my wheels. For the first iteration of this feature I decided to say to heck with streaming, I'll just wait until the whole test suite run is finished and send the logs in one big chunk.

In the spirit of "do the simplest thing that could possibly work", I sent the logs from the worker to the web server via a curl command using Basic HTTP authentication with hard-coded credentials. On the web side, I didn't use cloud storage, I just stuck the content right in the database, and then I crapped out the content to the screen and wrapped it in a pre tag. Did I expect to stick with all these solutions in the long run? Obviously not. I expected to end up rewriting literally all of it. The reason I started with these "stupid" solutions is that doing so allowed me to get the feedback cycle started early. Prior to when I saw logs in my browser for the first time, the answer to "When will we see something that works?" was "Who knows!" That's not a great position to be in. It reeks of uncertainty and risk. Once I had a very rough version of the feature in place, the question changed from "When will we see something that works?" to "When will we see something that works a little better?" It's a bit like the difference between sailing the open ocean, hoping to spot some land at some point, and sailing from island to nearby island. The chances of getting lost at sea are much smaller.

Here's what I had at this point:

  • Log content gets sent from worker to web server, in one single chunk, using curl and hard-coded Basic HTTP authentication
  • Log content gets saved to the main database by the web server
  • Log content gets shown in the browser in raw form

In any project I always try to work on the hardest and most mysterious work first. What in this picture was the most mysterious? It wasn't migrating from hard-coded Basic HTTP authentication to something more production-ready. It wasn't moving log storage from the database to the cloud. Nor was it formatting and styling the log output. The most mysterious part to me at that point was the actual streaming part H of the log streaming. How was I going to accomplish that?

My naive first guess at how real-time log streaming might work was something like the following (in pseudo-bash): 

while true; do
  LOG_CONTENT=$(cat log.txt)
  curl https://www.saturnci.com/api/v1/test_suite_run/123/logs $LOG_CONTENT
  sleep 5
done

This first pass doesn't try to be efficient. It reads in the entirety of the logs and then sends the whole thing off to the server. A nice part about doing it this way is that the server-side code doesn't even immediately have to change. It will just save to the database whatever log content it's given, and then I can repeatedly refresh the page to see the log content come through in real time. It's clunky as hell, and it's a far cry from what I ultimately want, but it's a small and un-risky piece of work which can get the feature meaningfully closer to the ultimate goal. As a matter of fact, even this absurdly shitty implementation of real-time log streaming is, in fact, real-time log streaming! The work from here on out doesn't consist of profound mystery-solving; it merely consists making the log streaming incrementally less crappy.

Time to again take stock of what we have:

  • Log content gets sent from worker to web server, the whole log file being sent all at once, but at 5-second intervals during the execution of the test suite run instead of just once at the end
  • Log content gets saved to the main database every 5 seconds by the web server (thanks to the worker hitting the web server every 5 seconds)
  • Log content gets shown in the browser in raw form and requires manual page refreshes in order to see current log content

What did I do next? Frankly, I don't remember, but I can tell you what I would do next, which is probably in fact what I did do next: make the browser fetch log updates automatically so that new log content shows up without having to refresh the page. My first solution to this was to use Turbo Stream broadcasts, but I later realized that this method was illogical and wasteful. The problem is that if the user has, say, three test suite runs running concurrently, then my web server broadcasts log updates for all three of those test suite runs every time an update comes in, even though the user is usually not going to be watching the real-time output of more than one test suite run at a time. For a test suite run whose logs the user is not watching, any log broadcast sent is a waste of resources.

The solution was simple: instead of making the web server say, "Hey, all you test suite runs in the browser, I have new log content for you!" I made the browser say, "Hey web server, I'm looking at test suite run X. Is there any new log content specifically for test suite run X?" To put it a different way, I switched from broadcasting to polling. Now a test suite run's log content would be sent if and only if someone had a browser tab open for that specific test suite run. Switching to polling had a double advantage: it was both conceptually simpler and significantly more performant.
