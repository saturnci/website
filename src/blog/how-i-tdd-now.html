How I TDD now

I'm loving TDD more than ever.

To state the obvious, agentic AI has caused a massive change to the programming profession over the last couple years, especially the last few months, and at this point almost nobody is writing code by hand anymore. (If you still are, my dear friend, I respectfully ask you what the hell you're doing with your life.)

In spite of all that has changed, which is a lot, one thing most certainly hasn't changed: the benefits of test-driven development.

To me, the core of TDD is what I call the specify-encode-fulfill loop: 1) decide on the specifications for what you want to build, 2) encode the first specification in the form of an automated test, then 3) write the code to fulfill that specification (then repeat from step 2). It's very similar to Kent Beck's Canon TDD workflow.

One of the many benefits of TDD is that it helps you separate what you want to do from how you want to do it, thereby lightening the cognitive load of programming and letting you work "at a lower RPM".

An even bigger benefit, and to me the main benefit of TDD, is that it leads to ~100% test coverage which gives me the ability to liberally refactor my code without the fear of introducing regressions. This allows me to improve the design of my code, which makes the code easier to understand and change, which allows me to work faster.

These benefits still apply when all the code is written by AI, which doesn't change the reality that the software we build is influenced by the feedback we get by putting it into production. Feedback always reveals to us things we wish we had done differently, and therefore refactoring will always be necessary. Test coverage is what enables that refactoring.

If you're interested in seeing the exact "skill" I use for doing TDD, here's my TDD skill on GitHub. I'll also just share the heart of it right here:

  1. Write a list of the specifications within scope of the current TDD session
  2. Encode one item in the list as an automated test
  3. Change the code just barely enough to make the current test failure go away. Avoid "speculative coding" — if we write more code than necessary to make the current test failure go away, we risk having code never exercised by any test
  4. Optionally refactor, but not before committing the behavior change. Never mix behavior changes with refactoring
  5. Until the list is empty, go back to #2

I've found that this workflow, combined with my test-design-review skill and the existing test examples in my codebase, results in pretty good tests, allowing me to practice the same TDD workflow I have for years, but even faster and more enjoyably now thanks to AI.
